# 面试准备笔记（AI Agent 方向实习）

> 本文是给项目作者自己的"弹药库"：高频问题的回答思路 + 简历写法。回答时结合代码位置说，含金量最高。

## 一、简历怎么写

**项目名**：AgentForge —— 企业级多智能体应用平台（个人项目，从 0 到 1）

参考条目（STAR，按需删减）：

- 从协议层自研 Agent 引擎（不依赖 LangChain）：设计消息/事件双协议与工具注册系统（函数签名 + docstring 自动生成 JSON Schema），实现支持并行工具调用、token/步数双熔断、错误自纠错回喂的 ReAct 循环；
- 设计事件溯源运行时：Agent 全过程以事件流输出并持久化，支持 SSE 断线续传（seq 去重 + Last-Event-ID 语义）、Checkpoint 断点恢复与 human-in-the-loop 高危工具审批；
- 实现中文友好混合检索：jieba 分词 + 自研 BM25（Okapi）与 pgvector 向量召回经 RRF(k=60) 融合，支持可插拔重排；实现 chunk 级引用溯源（答案 [n] 角标可回跳原文）；
- 构建深度研究多 Agent 流水线：规划 → 并行搜索子 Agent → 证据交叉验证 → 流式报告生成 → 评审修订，并行子 Agent 事件经异步流合并实时上浮；
- 建立评估体系：检索指标（Recall@K/MRR/nDCG）+ LLM-as-judge（忠实度/相关性/引用规范）+ 任务完成率，CLI 一键出报告并落库；
- 工程化：110+ 个离线确定性测试（自研 Mock Provider 可按 JSON Schema 生成合法结构化输出）、mypy 全量类型、带指标阈值的 GitHub Actions CI、Alembic 迁移、单镜像部署、全链路 Trace。

**一句话定位**（自我介绍用）：“我做了一个不依赖框架、从协议层自研的企业级 Agent 平台，重点解决了生产落地最关心的四件事：可观测、可评估、可控（HITL）、可恢复。”

## 二、高频问题与回答思路

### Q1：为什么自研，不用 LangChain / LangGraph？

分三层答：
1. **目的**：这是深度学习型项目，自研才能真正理解 Agent 循环、上下文管理、流式架构的每个细节；用框架三天能搭完，但面试讲不出东西。
2. **技术判断**：LangChain 抽象层次多、调试黑盒；LangGraph 的核心价值（状态图、checkpoint、事件流）我针对自己的场景做了更薄的实现——事件协议、检查点格式、追踪粒度完全自主可控。
3. **不排斥框架**：工具 Schema 与 OpenAI 规范一致、Supervisor/Planner 与 LangGraph 的范式一一对应，迁移成本很低；生产选型我会按团队栈来。

### Q2：ReAct 循环里最难处理的是什么？

- **并行工具的事件流**：一批 tool_calls 并行执行时，过程事件要实时上浮（否则前端卡住），但回喂模型的 tool 消息必须保持原顺序（协议要求）。我用 asyncio.Queue 合流事件 + 按 call id 归位结果解决（`core/agent.py` `_execute_tools`）。
- **失败不中断**：未知工具/参数非法 JSON/超时/异常，全部转成错误文本回喂，让模型自己纠错，而不是抛异常杀掉整轮对话。
- **熔断后的收尾**：达到步数/预算上限时不能硬停，要注入系统提示强制"基于已有信息作答"并摘掉 tools，保证用户总能拿到答案。

### Q3：上下文（记忆）怎么管理？

三级：① 系统提示（人设+工具规则+长期记忆注入）；② 短期：token 预算内保留原文，超出则把旧轮次滚动压缩成摘要（与旧摘要合并，保留关键信息清单）；③ 长期：LLM 结构化抽取事实 → 向量化 → 相似度去重 → 跨会话召回。再加一句：估算 token 用 CJK 0.6/字的经验公式，精确值以 API usage 为准。

### Q4：SSE 断线了怎么办？事件会丢吗？

事件溯源设计：每个事件带递增 seq 持久化；订阅时"先挂实时队列 → 再重放历史 → 实时流按 seq 去重"，配合 `?after=` 参数实现续传。瞬态的 llm_delta 不落库（有全量事件兜底），防止 DB 膨胀。这套设计同时解决了"刷新页面还能看到执行过程"的需求。

### Q5：HITL 审批是怎么实现的？如果用户一直不点呢？

工具标记 `requires_approval` → 循环执行前 yield `ApprovalRequired` 并 await 审批门（asyncio.Future）→ 运行状态置 `awaiting_approval` → API 决策后 resolve。超时 10 分钟自动拒绝。拒绝不是终止：把"用户拒绝"作为工具结果回喂，模型会换路径或直接说明。

### Q6：中文 RAG 和英文比要注意什么？你的检索为什么是这样设计的？

- 分词：BM25 必须建立在合理分词上，我用 jieba 搜索粒度 + 停用词表；
- 混合召回：向量管语义、BM25 管精确词（产品型号、金额、条款编号这类中文企业文档高频查询），RRF 融合不需要调权重（k=60 经验值）；
- 分块：标题路径感知（"手册 > 部署 > 环境"存进 heading），中文按句号/问号切句，块间重叠防止边界信息丢失；
- 溯源：不能只给答案，企业场景必须能点开 [n] 看原文，否则没人敢用。

### Q7：怎么评估你的 Agent / RAG 好不好？

- 检索层：标注 query → 相关文档，算 Recall@K/MRR/nDCG（确定性、便宜、能回归）；
- 生成层：LLM-as-judge 按忠实度/相关性/引用规范打 1-5 分（判官模型与被测模型分开配置）；
- Agent 层：任务完成率 + 平均步数/tokens/成本（效果与效率一起看）；
- 工程上：数据集 JSONL 管理、CLI 一键跑、报告落库可对比版本回归。诚实说明：样本量小，是"回归护栏"而非 benchmark。

### Q8：成本怎么控制？

预算熔断（token_budget）、按模型定价表核算每个 Span 成本并在 Trace 页可视化、便宜模型跑判官/摘要等辅助任务、检索先行减少长上下文、研究流水线限制并行度与来源数。

### Q9：沙箱安全怎么做的？

当前：`python -I` 子进程 + 临时目录 + 超时强杀 + 输出截断 + 默认人工审批；坦承局限：没有 syscall 级隔离，生产要上容器（gVisor/Firecracker），我的执行器接口已经抽象好，替换成本低。加分点：web_fetch 做了 SSRF 防护（解析后拒绝内网 IP）。

### Q10：并发模型讲一下？为什么全用 asyncio？

LLM 应用是典型 IO 密集：等模型、等搜索、等 DB。全链路 async（FastAPI + httpx + SQLAlchemy async + asyncpg），并行点在：一步内的多工具、多搜索子 Agent、评估批量任务。CPU 密集的极少数点（jieba 首次加载、文件解析）走 `asyncio.to_thread`。单 worker 内事件总线用队列；多 worker 靠 Redis Pub/Sub（预留接口）。

### Q11：如果压测/扩展到多实例，瓶颈在哪？

诚实版：① 事件总线是进程内的 → 换 Redis Pub/Sub；② BackgroundTasks 入库 → 任务队列；③ BM25 索引在内存 → ES；④ SQLite 写并发差 → 已默认建议 pg；⑤ LLM API 才是真正延迟大头 → 限流/排队/降级模型策略。能主动说出瓶颈和方案，比说"我都做了"更可信。

### Q12：Supervisor 和工作流两种编排怎么选？

自主委派（Supervisor）灵活但不可控、成本不可预算，适合开放任务；工作流（Planner-Executor/研究流水线）阶段确定、可并行可重试，适合结构化交付物。我的判断：**生产里 80% 场景应该用工作流，把自主性留给每个节点内部**——这也是业界共识（Anthropic 的 building effective agents 一文同观点）。

## 二·B、进阶能力问答（增强版）

### Q13：讲讲你对 MCP 的理解，你是怎么接的？

MCP（Model Context Protocol）是 Anthropic 提出的"Agent 与外部工具/数据源"的标准协议，基于 JSON-RPC 2.0，核心三步：`initialize` 握手 → `tools/list` 发现 → `tools/call` 调用。我实现了一个 MCP 客户端：抽象了传输层（stdio 子进程 / 进程内两种），连接后把发现的工具**包装成我自研引擎的 Tool 对象**（名字 `mcp__<server>__<tool>`，schema 直接透传 inputSchema），于是 ReAct 循环能像调用本地工具一样调用 MCP 工具。还写了个零依赖的示例 stdio server。价值：一次对接，生态里所有 MCP server（文件系统、数据库、GitHub 等）都能即插即用。

### Q14：语义缓存怎么设计的？会不会返回过期/串味的答案？

缓存键包含 **user + agent revision + KB revision + LLM + embedding model**，知识库或 Agent 变化会自然切换版本；TTL 和 0.93 余弦阈值控制复用。含“刚才/第二个/现在/最新”等上下文或时效词的查询直接绕过缓存。同一轮预先计算一次 query embedding，缓存查询、写入和长期记忆复用，避免三次重复请求。

### Q15：安全护栏具体防什么？

分输入、输出两道：**输入**——Prompt 注入/越狱检测（中英文规则打分，命中阈值拦截）+ 内容审核（违法/暴力等类别）；命中就拒答，不进入 Agent。**输出**——PII 脱敏（手机号/身份证/邮箱/银行卡/IP 正则识别并打码）。我把它做成独立引擎，纯函数、可测、可开关，并留了 hook——生产可换成专用模型（Llama Guard）或云审核 API。注入检测是启发式，我会坦白它不是万能的，深度防御还需要权限最小化 + 工具审批（我的 HITL）。

### Q16：RAG 进阶都做了什么？各自解决什么问题？

四个可开关的增强：① **查询改写**——把口语化/带指代的问题改写得更适合检索；② **HyDE**——生成"假设答案"用于向量检索（假设答案比问题更接近真实文档，且我做了向量用 HyDE、BM25 仍用原词的分离设计）；③ **上下文压缩**——逐块抽取与问题相关的句子，砍掉噪声降 token；④ **父子分块（small-to-big）**——用小块检索保证精度，命中后扩展到相邻块补全上下文。默认只开父子分块（不额外花 LLM 调用），其余按需开。

### Q17：可观测性除了 Trace 还做了什么？

加了聚合看板 + Prometheus。看板聚合近 N 天的运行数/成功率/token/成本/延迟趋势、工具使用 Top、缓存命中率、系统能力总览。同时暴露 `/api/dashboard/metrics` 的 Prometheus 端点（runs/tokens/cost/duration/tool_calls/cache/guardrail 等指标），可直接接 Prometheus + Grafana 做生产监控告警。这体现"从 demo 到生产"的意识。

### Q18：你怎么证明引用可信，而不是模型随便写了一个 [1]？

我把“编号存在”和“事实被证据支持”拆开。确定性 `CitationAudit` 检查无效编号、事实句引用覆盖率、来源是否读取过原文；前端只把真实存在的编号渲染成引用角标。来源注册表在服务端保留证据正文，研究评审模型会同时看到报告与证据，而不只是编号列表。确定性门失败会强制修订；达到上限仍失败则标记 `needs_review`，报告可查看但不能公开分享。这样不会把 LLM-as-judge 当唯一真相。

### Q19：服务重启、数据库迁移和扩容是怎么处理的？

生产镜像先跑 `alembic upgrade head`，应用进程不再 `create_all`；`readyz` 会校验数据库和 Alembic 版本；迁移链做过 `upgrade→downgrade→upgrade` 往返自测。崩溃遗留的 `running/awaiting_approval/resuming` 会在启动时收敛为 `interrupted`，chat 可基于 checkpoint 手动恢复并用 CAS 防重复（恢复启动若失败会把状态归还 `interrupted`，不会永久卡死），研究任务不自动重放副作用。当前事件总线和审批所有权是进程内实现，所以明确限制单副本单 worker，而不是声称已经支持水平扩展；后续要扩展会把任务所有权和事件总线一起迁到分布式队列。

### Q20：SSRF 和缓存串味这些边界你怎么防？

web_fetch 与自定义 HTTP 工具只放行全局可路由地址（`ip.is_global` 一条规则同时挡住私有段、环回、链路本地含云元数据 `169.254.169.254`、CGNAT `100.64/10`），并禁用自动重定向防止 3xx 绕过；彻底防 DNS 重绑定还需网络层出口白名单，这点我会诚实说明是部署侧职责。缓存方面：键含 user/agent/KB/模型多版本，时效或指代类查询正则绕过，且**注入了长期记忆的个性化回答不写入共享缓存**（会随用户画像演化而过期），引用不完整的回答也不缓存。引用审计还堵了一个隐患——要求引用时若报告根本没有来源会直接判失败，避免"无源报告"被误标 `succeeded` 并公开分享。
